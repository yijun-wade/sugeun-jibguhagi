## 1. 자연스러운 유저 경험을 위한 지역 검색 설계

사용자가 `창동 레미안`이라고 검색할 때, 결과는 단지명 일치뿐 아니라 사용자가 기대하는 지역적 맥락까지 반영해야 한다. 예를 들어 `레미안` 단지는 다음 순서로 노출되는 것이 자연스럽다.
1. 창동에 있는 레미안
2. 창동과 가까운 지역(예: 노원구)의 레미안
3. 서울 내 다른 지역의 레미안
4. 부산 등 먼 지역의 레미안

또는 검색 의도에 따라 다음 구성이 더 적절하다.
1. 창동의 레미안
2. 창동 주변의 다른 아파트 단지들

## 2. 검색의 러프한 처리 순서

1. 사용자가 검색어를 입력한다.
2. 검색어를 분석해 지역 키워드를 추출한다.
3. 추출한 키워드를 실제 지역 엔티티에 매핑해 polygon 또는 지역 코드를 얻는다.
4. (3)의 지역 범위와 원래 검색어로 텍스트 유사도 검색을 수행한다.
5. 결과가 부족하면 인접 지역 또는 상위 지역으로 범위를 확장한다.

요약하면 `검색어 분석 -> 지역 범위 추출 -> 검색 수행`의 3-step이다. 지역 계층은 시(서울시)/구(도봉구)/동(창동)의 polygon 또는 지역 코드로 해석하고, 좁은 범위일수록 높은 가중치를 부여해 창동 → 도봉구 → 서울시 순으로 노출되도록 점수를 설계한다.

## 3. 지역 코드와 좌표/폴리곤 기반 검색의 선택

지역을 어떻게 표현·비교할지 먼저 정해야 한다. 크게 지역 코드 방식과 좌표/polygon 방식으로 나뉜다.

가장 단순한 방식은 지역 코드다. 단지 데이터에 시/구/동 코드가 저장돼 있으면, 검색어의 지역 표현을 코드로 해석한 뒤 해당 코드 범위를 우선 검색한다. 구현이 단순하고 빠르며 행정구역 단위 처리, 데이터를 미리 전처리 할 수 있는 경우에 적합하다.

좌표와 polygon은 실제 공간상 위치와 범위를 다루며, 행정적 포함 관계 대신 거리·방향·범위 표현에 적합하다. 기본은 지역 코드로 처리하되, 다음 기능이 필요해지면 좌표/polygon을 함께 관리한다.

- 현재 리소스와 가장 가까운 다른 리소스 조회
- 넓은 행정구역 내부에서의 세부 거리 비교
- 인접 지역 확장의 정교화 (왼쪽으로 가까운지, 오른쪽으로 가까운지) 
e.g) 서울과 가장 가까운 행정구역은 경기라 했을때, 은평구는 고양(왼쪽 경기도)이랑 가깝지만, 그냥 "인접 시"로 비교하면 판교 (아래쪽 경기)의 리소스가 반환될 수 있음.
- 지도 기반 검색 또는 반경 검색

## 4. 필요한 데이터

아래 부터는 poligon보다는 데이터전처리와 텍스트를 통한 검색 준비 과정을 다룬다.
(제가 폴리곤을 잘 몰라요 )

### 4.1 검색어 분석용 데이터

사용자 입력에서 지역 키워드를 추출해 실제 지역 정보와 연결한다. `창동 레미안`, `도봉구 신축 아파트`, `서울 잠실 리센츠`에서 `창동`, `도봉구`, `서울`, `잠실`을 식별한다.

alias는 동일 표현뿐 아니라 해당 지역을 떠올리게 하는 다양한 표현(학교명·역명·랜드마크·생활권 등)을 담는다. 기준은 오타 보정이 아니라 "이 검색어가 들어오면 이 지역을 먼저 떠올릴 수 있는가?"다.

예시 필드: 지역명 / 지역 타입(시·구·동) / 지역 코드 / parent_gu_code / parent_si_code / aliases.

```json
[
  {
    "name": "창동",
    "type": "동",
    "code": "changdong",
    "parent_gu_code": "dobong",
    "parent_si_code": "seoul",
    "aliases": ["창동", "창동역", "창일중", "창일초"]
  }
]
```

이 데이터로 지역을 동 단위로 해석하고, `창일중 근처 아파트`처럼 동명이 직접 포함되지 않은 검색어도 해당 동과 연결한다. `parent_gu_code`·`parent_si_code`로 상위 확장과 가중치 계산도 처리한다.

### 4.2 실제 위치 데이터

아파트/단지 검색에 필요한 최소 필드: 이름, 주소, 시 코드, 군/구 코드, 동 코드.

```json
[
  {
    "name": "창동 레미안 1차",
    "address": "서울특별시 도봉구 창동 123",
    "si_code": "seoul",
    "gu_code": "dobong",
    "dong_code": "changdong"
  },
  {
    "name": "상계 레미안",
    "address": "서울특별시 노원구 상계동 456",
    "si_code": "seoul",
    "gu_code": "nowon",
    "dong_code": "sanggye"
  }
]
```

좌표 정보는 polygon 기반 검색이나 거리 계산이 필요한 경우에만 추가한다.

```
"lat": 37.6541,
"lon": 127.0472
```

## 5. 실제 쿼리 흐름

1. 검색어 분석 — 지역 키워드(`창동`)와 일반 키워드(`레미안`) 분리
2. 지역 키워드 추출
3. 지역 코드 해석 — 시(서울시), 구(도봉구), 동 코드 확보
4. 시/구/동 조건을 `where in`으로 함께 적용해 후보군을 넉넉하게 조회 (예: 상위 300개)
5. 가장 좁은 범위 매칭 결과부터 우선 노출 (창동 → 도봉구 → 서울시)

Elasticsearch는 텍스트 유사도와 지역 우선순위를 하나의 score로 합쳐 정렬할 수도 있으나 고수용이라 여기서는 생략한다. 관련 키워드: scoring_script, nori analyzer, filter.

## 6. 데이터 처리 주의 사항과 팁

**1. 공공 데이터 형식은 불균질하다.** 시/군/구 데이터가 일관되게 쓰이는지, 빈 값이 과도하지 않은지 먼저 확인한다. 표기가 중구난방이면 Naver 또는 Kakao 지도 API로 주소·지역 표현을 보강한 뒤 서비스 스펙에 맞게 정규화해 저장한다. 원본을 그대로 신뢰하지 말고, 검색에 필요한 최소 단위(시/군/구/동) 표현의 안정성을 먼저 점검한다.

**2. 데이터 적재 과정은 자동화한다.** 전체 흐름: (1) 공공 데이터 호출 → (2) Kakao API 등으로 지도 데이터 보강 → (3) 우리 DB 스펙에 맞게 변환 → (4) Elasticsearch indexing. 각각을 Claude Code skill로 만들고 이를 한 번에 실행하는 planner skill을 둔다. 필요 시 autoresearch의 reason 기능으로 중간 판단·보강 단계를 보조 처리한다.

## 추가 조언 — 실제 데이터 보고 확인해볼 것들

이 설계는 전반적으로 방향이 맞는데, 실제 데이터로 가기 전에 세 가지만 짚어두고 싶다.

### 1. 좌표계 (polygon/반경 기능 넣을 계획이면)

국내 공공 공간데이터는 EPSG:5179 (TM중부), 5186, 4326 (WGS84) 이 섞여 있다. 좌표계를 통일하지 않으면 거리/반경 계산이 수백m~수km 단위로 틀어질 수 있다. ""Elasticsearch의 geo_point/geo_shape는 4326 기준이므로, 적재 시점에 4326으로 재투영하는 파이프라인을 초반에 박아두는 게 좋다.

### 2. "정본 코드"는 용도별로 다름

§4.1의 `code` 필드를 한 체계로 통일하려 하면 나중에 막힌다. 한국 주소는 용도별로 식별자가 다르다.
- 매물/필지 식별: PNU 19자리
- 행정/통계 조인: 법정동 10 또는 8자리 (조인할 공공데이터 파일의 자리수에 맞춤)
- 검색 필터: 법정동 10 + 행정동 alias 병행 (사용자는 행정동 이름도 많이 씀)

어느 쪽을 정본으로 둘지는 조인할 공공데이터와 매물 원천 데이터 key를 먼저 보고 결정해야 한다.
